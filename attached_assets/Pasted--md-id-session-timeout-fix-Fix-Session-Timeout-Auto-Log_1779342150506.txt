# Fix Session Timeout / Auto Logout Logic Across the System

The current session timeout implementation is unreliable and behaving incorrectly.

There are 2 major issues:

1. Sometimes the session DOES NOT expire even after long inactivity.
2. Sometimes the session expires EVEN WHEN the user is actively working on the system.

This is a critical authentication/session management issue and must be fixed properly at the architecture level.

---

# CURRENT ISSUES

## Issue 1 — Session Not Expiring

Sometimes users remain logged in indefinitely even after:

- prolonged inactivity
- idle browser tab
- system inactivity
- no interaction

This is a security issue.

---

## Issue 2 — Session Expiring While User Is Active

Users are getting logged out even while:

- typing
- navigating pages
- clicking buttons
- using filters
- scrolling
- working continuously

This is incorrect and creates a very poor user experience.

---

# REQUIRED BEHAVIOUR

# SESSION TIMEOUT RULES

## ACTIVE USER

The session MUST remain alive while the user is actively using the application.

ANY valid user interaction should refresh/reset the inactivity timer.

---

## INACTIVE USER

The session should ONLY expire after:

```txt
X minutes of TRUE inactivity
```

Example:

```txt
30 minutes of inactivity
```

---

# REQUIRED USER ACTIVITY DETECTION

The inactivity timer MUST reset on:

```ts
mousemove
mousedown
click
scroll
keypress
keydown
touchstart
touchmove
pointermove
visibilitychange
focus
navigation
API activity (optional)
```

---

# IMPORTANT

DO NOT rely ONLY on:

```ts
mousemove
```

or

```ts
setTimeout
```

without proper cleanup/reset logic.

---

# REQUIRED ARCHITECTURE

# USE CENTRALIZED SESSION MANAGER

Create ONE reusable global session timeout manager.

DO NOT implement timeout logic separately in different pages/components.

Use:

- Context Provider
OR
- Global Auth Provider
OR
- Central Session Hook

---

# REQUIRED IMPLEMENTATION

## Recommended Flow

### 1. User activity detected

```ts
resetInactivityTimer()
```

---

### 2. Timer restarts cleanly

```ts
clearTimeout(existingTimer)
```

then:

```ts
startNewTimer()
```

---

### 3. No activity for configured duration

```ts
logoutUser()
```

---

# CRITICAL BUG TO FIX

Likely current issue:

- multiple timers running simultaneously
- timers not cleared correctly
- stale closures
- event listeners duplicated
- timers re-created on every render
- background tab issues
- visibility state mishandling

---

# REQUIRED TIMER HANDLING

## Correct Example

```ts
let inactivityTimer: NodeJS.Timeout;

const resetTimer = () => {
  clearTimeout(inactivityTimer);

  inactivityTimer = setTimeout(() => {
    handleSessionTimeout();
  }, SESSION_TIMEOUT_MS);
};
```

---

# REQUIRED EVENT CLEANUP

Ensure ALL listeners are cleaned properly:

```ts
return () => {
  events.forEach(event =>
    window.removeEventListener(event, resetTimer)
  );

  clearTimeout(inactivityTimer);
};
```

---

# BACKGROUND TAB HANDLING

## REQUIRED

Handle browser tab visibility properly.

When tab becomes active again:

```ts
document.visibilityState
```

validate:

- session expiry
- last activity timestamp
- token validity

# TOKEN VALIDATION

Session timeout must sync with:

- JWT expiry
- backend session expiry
- refresh token expiry

Frontend and backend timeout durations MUST match.
