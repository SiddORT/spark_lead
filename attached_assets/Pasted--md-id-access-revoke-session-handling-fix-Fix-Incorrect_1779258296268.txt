# Fix Incorrect Logout & "Email Not Whitelisted" Message After Access Revocation

There is a major session and authorization handling issue in the system.

---

# CURRENT INCORRECT BEHAVIOR

Scenario:

1. User is logged into the system
2. User is currently on a page (example: Analytics)
3. Admin revokes ONLY one permission/access right
4. User immediately gets logged out
5. System shows:

```txt
Your email ID is not whitelisted
```

6. System shows:
- Redirect to Login
- Request Access

This behavior is COMPLETELY WRONG.

---

# WHY THIS IS WRONG

The user:

✅ STILL EXISTS in the system  
✅ IS STILL WHITELISTED  
✅ STILL HAS A VALID ACCOUNT  
✅ STILL HAS VALID LOGIN ACCESS  

ONLY a specific module/page permission was revoked.

This is NOT an authentication failure.

This is ONLY an authorization/permission update.

---

# REQUIRED CORRECT BEHAVIOR

If a user loses access to a specific page/module while actively logged in:

## The system MUST:

✅ Keep the user logged in  
✅ Preserve the active session  
✅ Recalculate permissions dynamically  
✅ Prevent access ONLY to the revoked module  
✅ Redirect user safely to an allowed page  
✅ Show proper permission message  

---

# REQUIRED NEW FLOW

## Example Scenario

User is on:

```txt
/analytics
```

Admin revokes:

```txt
analytics.view
```

---

# EXPECTED SYSTEM BEHAVIOR

The system should:

1. Detect updated permissions
2. Check whether current route is still allowed
3. If route is no longer permitted:
   - redirect user to dashboard/home
4. Show message:

```txt
Your access to Analytics has been revoked by the administrator.
```

NOT:

```txt
Your email ID is not whitelisted
```

---

# CRITICAL DISTINCTION

## Authentication vs Authorization

### Authentication Failure

Occurs when:
- user deleted
- account disabled
- whitelist removed
- token invalid
- session expired

ONLY in these cases should logout occur.

---

### Authorization Failure

Occurs when:
- one module permission revoked
- role updated
- access scope changed

In these cases:
❌ DO NOT LOG OUT USER  
❌ DO NOT SHOW WHITELIST ERROR  
❌ DO NOT SHOW REQUEST ACCESS FLOW  

Instead:
✅ update accessible routes dynamically

---

# REQUIRED IMPLEMENTATION

## Backend

Separate these states clearly:

```ts
{
  isAuthenticated: true,
  isWhitelisted: true,
  permissions: [...]
}
```

DO NOT combine:
- whitelist validation
- permission validation

into the same failure response.

---

# REQUIRED API RESPONSE CHANGES

## WRONG

```json
{
  "success": false,
  "message": "Email not whitelisted"
}
```

---

## CORRECT

### Case 1 — Permission Revoked

```json
{
  "success": false,
  "type": "PERMISSION_REVOKED",
  "message": "Access to this module has been revoked"
}
```

---

### Case 2 — Actual Whitelist Removal

```json
{
  "success": false,
  "type": "ACCOUNT_DISABLED",
  "message": "Your account access has been revoked"
}
```

ONLY this case should trigger logout.

---

# FRONTEND REQUIRED FIX

## Route Guard Logic

Current logic likely does:

```ts
if API fails:
  logout user
```

This is WRONG.

---

# REQUIRED NEW LOGIC

```ts
if (error.type === "PERMISSION_REVOKED") {
  redirectToSafePage();
  showPermissionToast();
  return;
}

if (error.type === "ACCOUNT_DISABLED") {
  logout();
  redirectToLogin();
}
```
